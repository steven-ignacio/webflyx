Workflow

Solo

1. Make changes to files
2. git add . (or git add <files>)
3. git commit -m "message describing the changes"
4. git push origin main

Team
1. Update my local main branch with git pull origin main
2. Checkout a new branch for the changes I want to make with git switch -c <branchname>
3. Make changes to files
4. git add .
5. git commit -m "a message describing the changes"
6. git push origin <branchname> (I push to the new branch name, not main)
7. Open a pull request on GitHub to merge my changes into main
8. Ask a team member to review my pull request
9. Once approved, click the "Merge" button on GitHub to merge my changes into main
10. Delete my feature branch, and repeat with a new branch for the next set of changes

Gitignore patterns

* - Matches any number of characters except a slash. Used to ignore all .txt files

	*.txt
/ - anchored to the directory containing the .gitignore file. This would ignore a main.py in the root directory, but not in any subdirectories.

	/main.py

Negation
You can negate a pattern by prefixing it with an exclamation mark (!). For example, to ignore all .txt files except for important.txt, you could use the following pattern:

*.txt
!important.txt

--------------------------------------------------------
Git Stash

-saves staged, or unstaged, but NOT COMMITTED changes to a separate storage

git stash

// stash with a message
git stash -m "jdsl v0 almost working"

// outputs contents of stash
git stash list

// takes out and re-applies the last added change in the stack
git stash pop

// applies the most recent stash changes without removing the stash from the list
git stash apply

// removes the most recent stash without applying the changes to the directory
git stash drop

//allows referencing of the stashes' index
# Apply the third (0, 1, 2) most recent stash
git stash apply stash@{2}

# Remove the third most recent stash
git stash drop stash@{2}

Git Bisect

uses binary search to narrow down the commit that introduced a certain change

git bisect start

git bisect good <hash> // marks a commit as good

git bisect bad <hash> // marks a commit as bad

git bisect <good/bad> HEAD // marks current commit

git bisect run ./scripts/bisect.sh

git bisect reset // exits bisect mode
